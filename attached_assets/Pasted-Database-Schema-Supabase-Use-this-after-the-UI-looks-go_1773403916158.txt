Database Schema (Supabase)
Use this after the UI looks good and you are ready to connect a real backend.
> Act as an Expert Supabase & PostgreSQL Architect.
> We are continuing to build StartID, the capital-focused startup discovery platform. Currently, the UI is built with mock data. Now, we need to design the database schema using Supabase.
> Core Requirement:
> The database must strictly support the 4 user roles (Startup, Investor, Service Provider, Enthusiast) and keep capital flow as the central entity.
> Please write the SQL schema (tables, relationships, and Row Level Security policies) for the following:
>  * Users Table: Extending Supabase Auth to include a role enum (STARTUP, INVESTOR, PROVIDER, ENTHUSIAST).
>  * Startups Table: The core entity. Needs fields for slug, name, one-line pitch, stage, industry, location, founded_year, target_raise, committed_amount, valuation, round_type, etc.
>  * Startup_Metrics Table: Linked to Startups (MRR, ARR, growth_percent, CAC, burn, runway).
>  * Startup_Team Table: Linked to Startups (name, role, linkedin_url, bio).
>  * Investors Table: Linked to Users (firm_name, check_size_range, stage_focus, industry_focus).
>  * Interactions Table: A polymorphic table to handle state between users and startups. Types should include: UPVOTE, BOOKMARK, EXPRESS_INTEREST, DATA_ROOM_REQUEST.
> Ensure you include basic Row Level Security (RLS) policies. For example: Startups can edit their own profiles, but anyone can read public startup profiles. Gated documents should only be accessible if an Interactions row exists where DATA_ROOM_REQUEST is APPROVED.
> Provide the SQL setup script and explain the relationships briefly.
